claude-kit · implementation notes

explain 스킬을 HTML 아티팩트로 대체

기존 텍스트 기반 explain을 두 갈래의 Anthropic 디자인 HTML 아티팩트로 분리한다 — 일반 explain(에세이)과 explain-code(기술 문서).

01스펙 해석

사용자 명령어: /claude-kit:implement claude-kit에는 내가 만든 스킬들이 있어. 그 중에 explain을 html을 만드는 스킬로 대체할거야. 근데 A, B가 있다고 했으니 B를 기본 explain으로 대체하고 A버전을 explain-code로 만들거야.

이전 세션의 맥락에서 두 스타일이 식별되어 있었다:

  • A — 기술 문서(tech-doc) 스타일: 1080px 2컬럼 grid + 280px sticky sidebar(Key files, Gotchas), sans 15px 본문, 흰 카드, callstack walkthrough, 펼침 코드. 출처: 공식 갤러리 04-code-understanding.html 류.
  • B — 장문 에세이(essay) 스타일: 740px 단일 컬럼, serif 19px 본문, 드롭캡·푸트노트·인사이트 블록, 큰 헤딩, 학술 톤. 출처: tracing-thoughts-language-model 류 research blog.

사용자의 매핑: B → 기본 explain, A → explain-code. 일반 explain은 어떤 대상이든 에세이로 풀어내고, code-specific일 때만 tech-doc 사이드바 패턴을 쓴다는 자연스러운 분리.

02주요 결정

영역상태결정
스킬 분리 명명 DECIDED explain(B/에세이), explain-code(A/tech-doc). 사용자 지시 그대로. explain-essay 같은 후보 대신 generic explain이 에세이를 받는다 — "기본 explain은 자유로운 에세이"라는 위계.
SKILL.md 길이 DECIDED 다른 스킬(현 explain 10줄)보다 *훨씬* 길어진다. 디자인 토큰, 폰트, 레이아웃, 패턴, 시각 요소 정책, 함정 등을 모두 임베드. 이유: SKILL.md가 곧 prompt이고, 새 세션에서 이 한 파일만으로 6-iteration 수준 결과물이 나와야 함.
외부 갤러리 — plugin에 동봉 REVISED 당초 "self-contained 위해 갤러리 참조 안 함"이었으나 사용자 요청으로 뒤집음. 갤러리는 Apache 2.0 라이선스, claude-kit은 MIT — 호환됨. 1차 5개 → 2차 전체 20개 + index.html + LICENSE + README.md + ATTRIBUTION.md를 references/html-effectiveness/에 동봉(총 452KB). 두 차례 확장: 처음엔 내가 본 5개만 → 사용자가 "다른 파일들은?" → 전체 복사. ATTRIBUTION.md에 파일별 용도 매핑. 특히 16-implementation-plan.html은 implement 스킬에 직접 관련되어 누락하면 안 됐던 핵심.
한글 폰트 정책 DECIDED 두 스킬 모두 동일: Hahmlet (Google Fonts variable) for serif+sans, Monaco for mono. unicode-range로 한글만 Hahmlet, Latin은 시스템 폰트. Noto Sans KR·Nanum Gothic Coding은 명시적으로 제외.
시각 요소 정책 DECIDED SVG는 둘 다 허용하되 "본문 보강, 대체 금지" 원칙. 애니메이션은 한 곳만, getTotalLength() 기반. prefers-reduced-motion 처리 필수.
출력 파일 이름 DECIDED <target-slug>-explained.html 패턴. 현재 디렉터리에 저장 후 open으로 브라우저에 띄움.
버전 범프 DECIDED 1.3.1 → 1.3.2 → 1.3.3 → 1.3.4. 각각 patch bump (스킬 형식 변경 → references 동봉 → implement essay 테마 적용). API surface는 그대로(스킬 이름·인자 동일), 모두 내부 변경.
implement 스킬도 essay 테마 DECIDED 사용자 후속 요청: implement가 만드는 implementation-notes.html도 explain의 essay 테마(폰트 포함)와 일치하도록. skills/implement/SKILL.md에 핵심 토큰(색·폰트·tableNote/Aside/badge 패턴)을 임베드, 세부 패턴은 explain SKILL.md와 references에 위임 (DRY). 이번 노트 파일 자체가 그 결과의 *현물 견본*이기도 함.

03스펙에서 의도적으로 벗어난 곳

Deviation 1 — SKILL.md 분량

기존 explain SKILL.md는 11줄. 새 explain은 ~120줄, explain-code는 ~170줄. 왜: 디자인 시스템을 prompt에 임베드해야 첫 빌드에 좋은 결과가 나옴. 짧은 prompt는 6-iteration의 함정에 다시 빠짐. 사용자가 이 세션에서 명확히 "한 번에 완성"을 원했음.

Deviation 2 — 출력 형식: 마크다운 → HTML

기존 explain은 chat 응답으로 마크다운을 출력. 새 버전은 HTML 파일을 디스크에 쓰고 브라우저로 열어줌. 왜: 사용자의 명시적 요청 — "마크다운이 아니라 html로 만들어지길". 부작용으로 conversation 안에서 즉시 읽기는 어려워지지만, 시각적으로 풍부한 아티팩트를 얻음.

Correction — v1 초안의 실수

1차 초안에서 기존 explain의 콘텐츠 지침을 거의 버리고 디자인 시스템만 임베드한 SKILL.md를 작성했음. 사용자가 즉시 지적: "기존의 explain에 있던 내용을 완전히 버린건가? 내가 바란건 기존의 explain의 내용을 마크다운이 아니라 html로 만들어지길 바란거고 테마는 앤트로픽의 테마를 사용하길 바랬던거였어." 수정: 두 SKILL.md 모두 기존 explain의 콘텐츠 지침("architecture, structure, why decisions, lessons, bugs, pitfalls, how good engineers think, analogies, anecdotes, very engaging, not boring textbook")을 그대로 보존하고, 그 아래에 How to render 섹션을 분리해서 디자인 시스템을 얹는 구조로 재작성. 콘텐츠 = 무엇을 말할까, 렌더 = 어떻게 보여줄까 — 두 관심사 분리.

04고려한 대안과 기각 이유

대안 A: 단일 스킬 + 모드 플래그

explain --mode=essay|tech 식으로 한 스킬에 옵션. 기각: 사용자가 명시적으로 분리를 요청. argument 해석도 더 단순해짐 (target 하나만 받음).

대안 B: 외부 템플릿 파일 import

SKILL.md를 짧게 유지하고 templates/essay.html, templates/tech-doc.html을 디스크에 두는 방식. 기각: claude-kit은 SKILL.md 단일 파일 구조. 추가 파일은 setup 스크립트 변경 필요. 또 stylesheet 임베드도 단일 파일이 더 robust.

대안 C: HTML 안에 React/JSX

인터랙티브 컴포넌트(탭, 토글)를 더 쉽게 쓰려고. 기각: 갤러리 사례 모두 vanilla HTML+CSS+JS. zero-deps 원칙 유지. <details> 같은 native HTML 패턴으로 충분.

05오픈 퀘스천

Open 1 — explain-code 외 도메인 특화 분기?

"코드"외에 다른 도메인(시스템 아키텍처, 데이터 모델, API 명세 등)에도 tech-doc 스타일이 적용될까? 일단 explain-code 하나로 두되, 필요하면 후속 세션에서 explain-system 등 추가 분기 가능. 지금은 보류.

Open 2 — 다국어 페일오버

현재 디자인은 한글+영어 혼용을 가정. 일본어/중국어 컨텐츠에서 Hahmlet은 fallback이 어색할 수 있음. 현재 스코프 밖 — 사용자가 한국어 환경.

Open 3 — 이미지/스크린샷 처리

target이 외부 URL이거나 스크린샷 분석이 필요한 경우, agent가 browser/curl을 통해 fetch 해야 함. SKILL.md에 이 흐름을 자세히 적지는 않음 — agent의 일반 능력에 위임. 관찰만.

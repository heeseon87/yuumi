# `yuumi:review` — 설계 문서

**작성일**: 2026-06-06
**상태**: 워크플로 ①②(이해)만 SKILL.md에 구현 — ③리뷰·④지도·⑤출력은 의도적 보류(이해 단계를 먼저 완성한 뒤 하류를 붙인다). 이 문서의 §3.7·§4③~⑤·§5 지도·§9는 그 향후 단계의 설계로 보존.
**설치 후 명령**: `/yuumi-review`

---

## 1. 문제와 동기

남이 작성한 코드를 *내가* 리뷰하기 어렵다. 병목은 두 단계다:

1. **이해가 안 된다** — 코드를 이해하는 것부터 쉽지 않다.
2. 이해가 돼야 비로소 리뷰할 수 있다.

기존 도구로는 이 빈틈이 안 메워진다:

- `yuumi:explain` 은 "이해"는 도와주지만 리뷰로 안 이어지고, **가독성은 올려도 이해는 안 만든다** (후술).
- 빌트인 `code-review` 는 **AI가 대신 판단**해버려서, 정작 *내가* 리뷰하는 능력은 안 길러진다.

이 스킬은 둘 사이의 빈틈에 산다: **남의 변경을 내가 능동적으로 이해하고, 내가 판단하도록** 돕는다.

---

## 2. 정체성과 차별화

핵심 DNA: **`teach-me`의 능동 검증 + 리뷰 대상**. AI는 *채점자*지, 모범답안을 먼저 보여주는 사람이 아니다.

| | 이해 대상 | 판단 주체 | 산출 | 매체 |
|---|---|---|---|---|
| `explain` | 대상 **전체** 멘탈모델 | (판단 없음) | HTML 에세이 | 수동 읽기 |
| 빌트인 `code-review` | (이해 안 깔아줌) | **AI** | PR 코멘트 | 자동 |
| **`review` (신규)** | **변경 + blast radius**만 | **설계·의존성 = 나** / **세부구현 = AI** | 터미널 코멘트 + 누적 지도 | 능동 대화 |

`explain`처럼 무겁게 전체를 그리지 않는다. **"이 변경이 뭘 하려 하고, 뭘 건드렸나"**만 가볍게. 시각 자산은 `pretty`에서 직접 차용한다 (`../pretty/assets/`, `../pretty/references/`). `explain`도 `pretty`의 소비자일 뿐이므로 `review`도 동일하게 `pretty`를 직접 참조한다.

## 3. 핵심 원리 (설계를 관통하는 축)

### 3.1 가독성 ≠ 이해. 갭을 메우는 건 능동성.

`explain`/`pretty`가 올려주는 건 **가독성**(정보가 눈에 잘 들어옴)이지 **이해**(머릿속에 정확한 모델 형성)가 아니다. 잘 정리된 걸 *읽으면* "안다는 착각"(illusion of competence)만 생긴다. 진짜 이해는 스스로 인출·재구성·시뮬레이션할 때만 생긴다. → 이해 단계는 **수동 읽기(HTML)에 맡기면 안 되고, 능동 대화로** 만들어야 한다.

### 3.2 아는 것으로부터 확장 (고수준 → 저수준)

가장 빠른 이해는 "아는 것에 닻을 내리고 확장"하는 것이다 (Ausubel의 advance organizer / 스키마 이론). 그래서 이해는 **반드시 고수준 → 저수준**으로 내려간다:

- 도메인 언어(이미 아는 것) → 메커니즘 → 구체 코드(모르는 것)
- 각 스텝이 직전 스텝에 닻을 내려 끊기지 않는다.
- bottom-up(코드 라인부터)은 닻 없는 파편만 쌓인다 — `explain`이 안 수월했던 이유.

### 3.3 각 단위는 What → Why 순서

고수준→저수준 사다리와 프랙탈로 겹친다. *전체*도 What(뭘 바꿨나)→Why(왜)고, *각 단위*도 What→Why다. What은 "관찰 가능한 사실"(닻 내리기 쉬움), Why는 "그 위에 쌓는 해석"(닻이 있어야 이해됨). Why를 먼저 들으면 닻 없는 추상이 된다.

### 3.4 앵커링 차단 — "뭐냐는 주고, 맞냐는 숨긴다"

AI가 평가 판정("이거 문제야")을 먼저 말하면, 사용자는 "그래 보이네" 하고 도장만 찍는다 (앵커링). 리뷰 근육이 안 는다. 그래서:

- **이해 단계**: AI가 *서술 정보*(고수준 요약, What)를 먼저 주는 건 OK — 그건 앵커링이 아니라 **advance organizer(발판)**.
- **리뷰 단계**: AI가 *평가 판정*(맞냐)을 먼저 주는 건 **앵커링 → 차단**. 사용자가 먼저 판단한 뒤 공개.
- 같은 "AI가 먼저 말함"이라도 하나는 약(서술)이고 하나는 독(판정)이다. 이 선이 스킬의 정밀함이다.
- **적용 범위**: 앵커링 차단은 *사람이 판단하는* 설계·의존성 레벨(3.7)에만 건다. 세부 구현은 사람이 판단하지 않으므로 AI가 거기서 판정해도 앵커링이 발생하지 않는다.

### 3.5 낮은 충실도(balsamiq 정신)도 앵커링 차단 장치

정교한 다이어그램은 "다 끝났다"는 느낌을 줘서 그 자체가 앵커링이다. 낮은 충실도(balsamiq 정신: 거칠고·잠정적·본질만)는 "같이 따져보자"는 느낌이라 사고를 자극한다. ASCII 다이어그램이 이 정신을 그대로 담는다 — 오히려 더 거칠어서 앵커링을 더 덜 일으킨다. (미감이 아니라 기능을 빌린다.)

**시각화는 의무다 (1차 스모크에서 도출).** 첫 스모크 출력에 ASCII 시각화가 0개였다 — 프롬프트가 시각화를 "절제/예외"로 프레이밍하고 "한 문장 닻" 가드레일이 그림을 억제로 읽혀, 핵심 기능이 질식했다. 그래서 디폴트를 **"허락"에서 "의무"로 뒤집는다**: 머릿속 연산이 필요한 지점(순서·루프 내 상태 변화·분기 조합·영향 범위·이름은 비슷하나 다른 것)은 *거의 모두* ASCII로 그린다. 텍스트로 뭉개는 게 실패고, 안 그리는 게 정당화 필요한 예외다. 금지하는 것은 *산문 문단*이지 *그림*이 아니다. 하한선 하나만 유지 — 연산이 전혀 없는 곳엔 안 그린다(decoration 방지).

**시각화 범위는 자료구조에 그치지 않는다 (2차 스모크에서 도출).** 2차 스모크에서 에이전트가 BEFORE/AFTER 자료구조 트리는 그렸으나, 계층 이동·소속 관계는 다시 텍스트로 뭉갰다. 대상은 *그녀가 머릿속에서 재구성해야 하는 모든 것* — 계층 간 이동(무엇이 어디서 어디로), 소속·포함(무엇이 무엇 아래), 의존 방향, before/after 구조 비교를 포함한다. 설계 판단을 물을 때도 그 구조를 *먼저 그려* 질문이 그림과 함께 도착하게 한다 (sentences로 재조립시키지 않는다).

**시각화는 모든 영역에 + 위치 지도 + 맵=흐름 미러 (3차 스모크에서 도출).** 3차에서 구조 변화는 잘 그렸으나 문제 진술·게이트 판정·디테일 발견은 다시 줄글 폭탄이었다. 세 교정: ① 시각화는 자료구조뿐 아니라 *모든 영역*(문제·긴장·판정·발견 목록 포함). 산문은 그림을 잇는 *실*일 뿐 — 구조·관계·대조·순서·목록을 줄글로 carry하면 실패. ② 전체 지형(클러스터)을 먼저 세우고, 각 섹션이 *전체 리뷰 중 어디*인지 표시한다(브레드크럼). 위치를 안 보이면 그녀가 매번 머릿속에서 지도를 재구성해야 한다. ③ HTML 맵은 터미널 흐름을 *미러*한다 — 변경 시작점부터 리뷰가 걸은 순서대로 누적. 테마만 좋고 내용이 겉돌면(독립적으로 wander) 실패. 맵의 유일한 가치는 "뒤돌아봤을 때 방금 걸은 같은 길이 보이는 것".

### 3.6 What/Why 노드 가드레일 — 코어 루프가 `explain`으로 붕괴하는 걸 막음

각 단위 노드(3.3)는 두 방향으로 무너질 수 있다. 3.1·3.3·3.4는 따로 보면 완벽하지만 *겹치는 지점*에서 충돌한다. 이 두 가드레일은 코어 루프의 의미론이므로 구현 시 반드시 지킨다:

- **(a) What/Why는 "최소 발판"이지 설명이 아니다.** 에이전트가 노드마다 What 한 문단 + Why 한 문단을 *써주면*, 사용자는 도로 수동으로 읽고 있다 — 3.1이 "이해 안 생긴다"고 못박은 바로 그 상태. 스킬이 `explain`에 능동 코스프레만 입힌 꼴로 회귀한다. 에이전트의 단위별 What/Why는 **한 문장짜리 닻(스캐폴드)**에 그치고, 인지 작업의 본체는 **사용자의 능동 예측**에서 나와야 한다. *에이전트가 문단을 쓰고 있으면 이미 수동으로 퇴화한 것이다.* (3.2의 advance organizer는 작은 스캐폴드지 설명 그 자체가 아니다.)
- **(b) Why = 작성자 의도(descriptive)이지 correctness 판정(evaluative)이 아니다.** 3.3의 Why는 "작성자가 왜 이렇게 짰나"(서술)여야 한다. 이걸 "왜 이게 옳은가"(정당화·평가)로 쓰면, 이해 단계가 리뷰 판정을 미리 흘려서 3.4 앵커링 차단이 통째로 무력화된다 — 잎에 도착하기도 전에 "이거 맞다"가 심어진다. correctness 평가는 오직 ③ 리뷰 단계의 ㉯에서만.

### 3.7 리뷰 분업 — 검출 가능성으로 가른다

리뷰를 두 종류로 나눠 주체를 다르게 한다. 기준은 **"기계적으로 검출 가능한가 vs 맥락·판단이 필요한가"**다.

| 레벨 | 이해 주체 | 리뷰 주체 | 처리 |
|---|---|---|---|
| **설계·의존성** (고수준~중간) | 사람 (능동 티키타카) | **사람 주도 + AI 백스톱** | 앵커링 차단 루프 (㉮ 내가 먼저 → ㉯ AI 대조 → 채택/수정/거부) → ㉰ AI 백스톱 검토 |
| **세부 구현** (잎: 경계조건·null·타입·자원 누수 등) | AI (필요 시 사람 드릴다운) | **AI 온전히** | 설계 확정 후 AI가 자동 검출·판정 → 지도/코멘트에 누적 보고. 사람에게 일일이 채택 여부를 묻지 않음 |

근거: 세부 구현 버그는 기계적으로 검출 가능해 AI가 사람보다 잘하고 지치지 않는다. 반대로 "의존 방향이 맞나, 추상이 적절한가, 설계가 변경에 열려있나"는 맥락·판단이 필요해 사람이 해야 한다. 사람을 세부에 갈아넣으면 정작 판단이 필요한 고수준에 쓸 집중력이 고갈된다 ("피곤하고 놓친다"의 정체).

**사람이 보는 깊이**: 이해 사다리는 사람용으론 보통 "잎 바로 위"(설계·의존성)에서 멈춘다. 잎(세부)은 AI 영역이므로 사람은 거기까지 내려가지 않아도 된다. 단 설계 판단에 필요하면 **사용자 주도 드릴다운**("이 잎 좀 보자")으로 내려간다.

**설계 확정 게이트 (AI 백스톱)**: 사람이 설계·의존성에 "더는 이슈 없음"이라고 판단하면, AI가 설계 리뷰를 **한 번 더 검토**한다 (사람이 놓쳤을 수 있으므로). 이슈를 찾으면 알리고 → 사람이 판단 → 수렴까지 반복. 사람·AI가 클린 합의에 이르러야 세부 구현 리뷰로 넘어간다. 두 가지 이유: ① 백스톱이 사람 판단 *뒤*에 오므로 앵커링을 깨지 않는다(학습 보호 + 안전망 동시). ② 설계가 흔들리면 세부 구현은 다시 짜야 하므로, 설계 확정 전 구현 리뷰는 낭비다 — 게이트가 헛리뷰를 막는다.

**부수 효과**: ① 규모 부담이 줄어든다 — 사람은 고수준~중간만 보므로 잎이 많아도 길어지지 않는다 (§6 보강). ② 앵커링 차단을 설계·의존성 레벨에만 걸면 되어 적용 범위가 명확해진다 (3.4).

### 3.8 이해의 미시구조 — 요약 골격과 종료조건 (스모크에서 도출)

**일반 규칙은 "변경의 인지 형태(cognitive shape)를 골라라"다 — 아래 골격은 그중 *수리(repair)* 형태일 뿐, 모든 변경의 기본값이 아니다 (oracle 추상성 리뷰에서 도출).**

변경이 **기대 위반을 수리**할 때만 다음 골격을 따른다:

```
⓪ 주제   도메인 언어로 무엇에 관한 변경인가  → 스키마를 켠다(정보 도착 전에 켜야 작동)
① 목표   무엇이 필요/기대되는가              → 문제가 위반하는 기준
② 현실   현재 코드는 무엇인가                → 목표와 대비
③ 충돌   왜 양립 불가인가 = 메커니즘         → 진짜 문제
④ 해결   변경이 무엇을 하는가
```

- **문제 = 기대의 위반.** 목표(①)를 먼저 세워야 위반(②③)이 문제로 보인다. 기준 없는 "코드가 X를 한다"는 중립이라 "그래서?"에서 멈춘다.
- **충돌은 결과가 아니라 메커니즘까지.** "상충한다"는 결과다. *왜* 양립 불가인지(예: 상이한 요구 + 단일 자원 + 비가역 가공 → 하나 살리면 하나 죽음)를 도메인 지식으로 자명해질 때까지 쪼갠다.
- **세 가지 인지 형태 (oracle 추상성 리뷰에서 도출)**: 수리 골격을 *보편*으로 제시하면 — 단서를 달아도 — 미래 에이전트가 *모든* 변경을 "깨진 기대"로 취급해 지어낸 드라마(invented drama)를 만든다. 이게 가장 큰 과적합 중력 우물. 그래서 형태를 먼저 고르게 한다: **수리**(기대 위반 → 기준·현실·메커니즘·해소) / **보존**(같은 동작·정체성을 새 이름·위치·버전·래퍼로 → 불변식을 명명하고 매핑을 그림, 없는 충돌 만들지 말 것) / **추가**(새 표면 + 기존 시스템 어디에 부착되나 + 주변에서 안 변해야 할 것). 프레임의 목적은 *다음 코드 스텝을 예측 가능하게* 만드는 것이지 충돌 이야기를 강요하는 게 아니다.
- **과적합 방지**: 형태 예시(수리/보존/추가)를 화이트리스트로 굳히지 말 것 — 일반 원리("다음 스텝을 예측 가능하게")가 지배하고 셋은 *예시*다. 긴장 없는 변경은 한 줄로 끝내되, **낮은 드라마 ≠ 낮은 인지부하** (oracle): 20파일 기계적 이동은 충돌은 없어도 "무엇이 어디로/무엇이 그대로"의 *지도*가 필요하다. 5비트 강요 = manufactured drama = 이 스킬이 죽이려는 바로 그것 (anti-pattern).
- **라벨 노출 금지 (2차 스모크에서 도출)**: 골격은 글의 *뼈대*지 *제목*이 아니다. "주제:/기대:/긴장:/해소:" 같은 라벨을 문단마다 박지 말고 산문으로 자연스럽게 흐르게 한다. 라벨이 보이면 강의처럼 느껴지고, 순서는 *느껴져야* 하지 *명명되어선* 안 된다.
- **신호 대 잡음 + 메커니즘 깊이 (4차 스모크에서 도출)**: 두 지적이 *하나의 원리*다. ① **방법 서술(method-narration) 금지** — "먼저 ~를 못 박자", "한 발 확인하고 내려가자" 같이 *설명에 관한* 말은 잡음이라 가독성을 *낮춘다*. 라벨 금지의 확장: 골격 라벨뿐 아니라 *자기 방법의 중계*도 빼고, 그냥 내용을 직접 말한다(말은 이해 *그 자체*여야지 이해하는 *행위의 서술*이 아니다). ② **메커니즘은 펼치고 그린다** — 정작 문제의 메커니즘을 한 문장으로 뭉개면 "왜 문제였나"가 신뢰로만 남는다(§3.8 "결과가 아니라 메커니즘"의 재확인). 여긴 *유일하게 압축하면 안 되는* 곳: 자명해질 때까지 쪼개고 *시각화*한다. ③ 통합: **간결함은 잡음용이지 실질용이 아니다 — 길이는 개념의 난이도를 따라가야지 일률 최소화가 아니다.** (자책 메모: oracle F3로 "a problem has a shape too"를 지우며 *문제 메커니즘을 그려라*는 신호까지 날아갔다 → 인지-형태 안전 프레임으로 복원: 모든 변경이 문제다 ❌ → *문제가 있을 때* 그 메커니즘은 그릴 가치 큰 관계 ✅.)

**종료조건은 양방향이다:**
- **아래 (자명 바닥)**: 도메인 지식으로 "당연"해질 때까지 why를 내려간다. 사용자의 "모르겠어"를 기다리면 *너무 일찍 멈춘 것* — 스스로 바닥까지 간다.
- **위 (코드 경계)**: why 사슬은 어느 지점에서 코드를 떠난다. 출처별로 라벨하고 경계에서 추론을 멈춘다 — 🟢연역(코드/언어, 단언) / 🟡추론(코드 패턴, "추정" 명시) / 🔴의도(제품·도메인, 코드 밖 → PR/이슈 → 없으면 사용자 질문, confabulate 금지). 코드는 결정의 *결과*만 담고 *왜*는 안 담는다.

**두 종류 Why**: Why-동기(왜 필요했나 = 긴장, framing이라 맨 앞) vs Why-구현(왜 이 코드, What 다음). §3.3 "What 먼저"는 후자에만 적용 — 둘은 층위가 달라 충돌이 아니다.

**느슨하게 풀되 힘은 유지 (oracle 추상성 리뷰에서 도출).** 과적합을 덜어내다 보면 능동성의 *힘*까지 빠진다. 추상화와 별개로 다음 셋은 *더 단단하게* 박아야 한다:
- **능동 루프 = ledge 주고 멈춤(handoff).** "예측을 끌어내라"만으로는 에이전트가 "넌 X를 예상하겠지, 실제로 그래"라며 자문자답해 본인이 일을 해버린다. 규칙: 발판 하나 주고 **같은 호흡에 답하지 말 것** — "물음표 붙은 에세이"는 대화가 아니다.
- **skip ≠ 이해.** §3.7의 사용자 주도 스킵은 보호하되, load-bearing 스텝에 *수동적 끄덕임*("괜찮아 보여")만 오면 작은 예측·재진술을 한 번 요구하고 넘어간다. "looks fine"이 illusion of competence가 살아남는 바로 그 틈이다.
- **blast radius는 코드 밖+안 양방향.** diff에서 멈추지 말되, *뒤로*(작성자 흔적=의도)만이 아니라 *밖으로 코드를*(caller·callee·test·contract) 읽어 변경이 닿는 범위와 안 건드리겠다는 약속까지 본다. (이해 범위의 blast radius — 보류된 리뷰/판정 단계와는 별개.)

### 3.9 컨텍스트 공백 코멘트 (파이프라인 피드백)

🔴 의도가 코드·PR 양쪽 모두에 없으면 → **사용자에게 먼저 질문** → 사용자도 모르면 그것이 진짜 공백이다 → **"PR 본문에 명시 요망" 코멘트**(correctness 결함과 별개)를 낸다. 리뷰가 상류(구현/PR) 손실을 진단해 환류하는 품질 게이트 역할. 전체 맥락은 [컨텍스트 파이프라인 아키텍처](2026-06-06-context-pipeline-architecture.md) 참조.

## 4. 워크플로

```
입력(PR# / 브랜치 / diff / 파일 — 인자 보고 판단)
   │  의도(도메인 한 줄)는 PR 제목·본문·커밋·이슈에서 추출.
   │  코드만으론 의도 복원 불가 시 → 지어내지 말고 자백("의도 불명, 너 알아?")
   ▼
① 전체 요약 (터미널)
   도메인 언어로 What+Why를 요약 수준으로 먼저 — 큰 그림부터 심음
   ▼
② 이해 단계 — 터미널 티키타카 (고수준→저수준 사다리)
   쪼갤 수 있는 단위 하나씩:
     - 각 단위 What → Why 순서로 능동 대화
     - 발판 생긴 지점부터 "다음 예측해봐" 게이트 (매 스텝 강제 X)
     - 머릿속 연산 터지는 stuck point: ASCII 시각화 우선,
       ASCII로 진짜 못 그리는 것(공간/그래프 구조)만 가벼운 HTML
     - 사용자 주도 스킵 가능 ("이 덩어리 됐어, 넘어가")
   ▼
③ 리뷰 단계 — 레벨별 분업 (3.7), 설계 확정 게이트 후 구현
   ▸ 설계·의존성 (사람 주도 + AI 백스톱):
       ㉮ 내가 먼저 판단 ("이 의존/설계 어때 보여?")
       ㉯ 그제서야 AI 대조/초안 공개 → 채택 / 수정 / 거부
       ㉰ 내가 "더는 이슈 없음" → AI 설계 백스톱 검토 (내가 놓친 것 포착)
            ├ 이슈 발견 → 알림 → 내 판단 → 수렴까지 반복
            └ 사람·AI 클린 합의 ↓ (설계 확정 게이트 통과)
   ▸ 세부 구현 (AI 온전히): 설계 확정 후 자동 검출·판정
       → 지도/코멘트에 누적 보고 (사람에게 일일이 안 물음)
   ▼
④ HTML 지도 — background subagent로 병렬 누적 갱신
   각 섹션/단위 끝날 때마다 에이전트가 background subagent 자동 호출
   → 메인 티키타카 안 끊김, 메인 컨텍스트 오염 0
   → 채택된 리뷰 코멘트도 지도에 누적 (영구 기록, 스크롤백 소실 방지)
   → 사용자는 브라우저에서 새로고침으로 "뒤돌아보기"
   ▼
⑤ 출력 — 터미널에 채택 코멘트 정리 (file:line) + 지도에 누적
```

### 이해 트리와 리뷰가 만나는 곳

사람의 리뷰는 사다리의 **설계·의존성 노드**에서 일어난다 (잎이 아니라). 고수준에서 내려오며 이해하다 "이 의존/설계 맞아?" 지점에서 사람 리뷰 루프로 전환된다. 잎(세부 구현)에 닿으면 AI 리뷰로 넘어간다 (3.7). 이해 → 리뷰가 레벨마다 한 흐름으로 엮인다.

## 5. 매체 아키텍처

| 역할 | 매체 | 비고 |
|---|---|---|
| 이해/리뷰 대화 | **터미널 티키타카** | 능동성 = 이해 엔진. 흐름을 안 끊는 게 최우선 |
| stuck point 시각화 | **ASCII 우선** | balsamiq 정신. 흐름 안 끊김 |
| ASCII가 못 그리는 구조 | 가벼운 HTML (탈출구) | 공간/그래프 구조만 |
| 누적 지도 | **HTML (`pretty` 자산)** | "지도"로 강등 — 이해 책임 X, 길 안 잃는 배경 + 돌아보기 |

### 지도는 background subagent로 갱신

`fork`(`/branch`)는 검증 결과 **에이전트가 자동 실행 불가(사용자 수동)** + 일회성 분기라 "섹션마다 누적 갱신"에 안 맞는다. 대신 `Agent` tool + `run_in_background`:

- 에이전트가 *자동으로* 띄움 (사용자 수동 조작 불필요)
- 독립 컨텍스트 → 메인 오염 0
- 백그라운드 → 티키타카 안 끊김
- 매 섹션마다 호출 가능 → 누적 갱신 OK

지도 "참조"는 세션 분기가 전혀 필요 없다 — HTML 파일이므로 브라우저 새로고침이면 끝.

## 6. 규모 전략

"차근차근하면 모든 규모 OK"가 두 장치로 성립한다:

1. **사다리 구조** — 위쪽은 폭이 좁아 빠르게 전체를 잡는다.
2. **리뷰 분업 (3.7)** — 사람은 설계·의존성(고수준~중간)만 보고, 잎(세부 구현)은 AI가 맡는다. 그래서 **잎이 많아도 사람의 부담은 안 늘어난다.** 규모는 주로 잎의 수로 폭발하는데, 그 폭발을 AI가 흡수한다.

그 위에 **사용자 주도 스킵**("이 덩어리 됐어, 넘어가")으로 운전대까지 쥐게 한다 (트리아지 강제 X). → "차근차근 + 모든 규모 + 내가 통제" 셋 다 충족.

terminal-only 출력의 스크롤백 소실 위험은 **누적 지도가 영구 기록 역할**을 해서 해소된다.

---

## 7. fallback / 정직 규칙

- **의도 복원 실패**: 코드만으로 도메인 한 줄(사다리 뿌리)을 못 만들면 confabulate 금지, "코드만으론 의도 불명, 너 알아?"라고 자백. (틀린 뿌리는 "자신 있게 잘못된 이해"를 낳아 가독성≠이해보다 더 위험.)
- **예측 강요 금지**: 이해 초반엔 발판이 없으므로 예측을 강요하지 않는다. 발판이 생긴 지점부터만.
- **시각화 절제**: stuck point(머릿속 연산이 실제로 터지는 지점)에서만. 장식용 다이어그램 금지 (explain의 "draw only the stuck points" 차용).

## 8. 구현 시 주의 (writing-plans로 이월)

- SKILL.md는 프롬프트 한 장이다. 사다리 + What→Why + 예측 게이트 + ASCII/HTML + 앵커링 차단 + 규모 스킵 + background 지도 + 자백 규칙을 다 평면으로 얹으면 실행 에이전트가 일부를 흘린다. **핵심 루프는 크리스프하게, 나머지는 조건부 가지로.**
- 9번째 스킬이 된다. 버전 범프는 `.claude-plugin/plugin.json` + 모든 `skills/<name>/SKILL.md`(이제 9개) `version:` 동기화 + `skills.sh.json` 그룹 추가까지.
- frontmatter `name: yuumi:review` → 설치 디렉토리 `yuumi-review` → 명령 `/yuumi-review`.
- supporting 파일은 `skills/review/` 안에 둬야 installer가 나른다. 단 시각 자산은 `pretty`를 `../pretty/`로 참조 (중복 금지).

## 9. 미해결/추후 결정 (구현 단계에서)

- 지도 HTML의 구체 구조 (변경 트리 + blast radius + 채택 코멘트 누적을 어떤 레이아웃으로).
- background subagent에 넘길 컨텍스트 페이로드 형식 (현재 이해 상태 + 기존 지도 diff).
- 리뷰 포인트가 사다리 잎과 1:1인지, 한 잎에 복수 포인트 가능한지의 표현.

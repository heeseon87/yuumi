# 컨텍스트 파이프라인 — 생성에서 리뷰까지

**작성일**: 2026-06-06
**상태**: 아키텍처 비전 (여러 스킬을 관통하는 상위 설계)
**관련**: [`yuumi:review` 설계](2026-06-06-review-skill-design.md), 향후 `PR`/`Implement` 스킬

> 이 문서는 `yuumi:review`를 만들며 도출된, **리뷰보다 상위의 시스템 비전**이다. 리뷰 스킬 하나로는 닫히지 않는 영역을 기록해, 이후 PR 작성 스킬·Implement 스킬(현재 것의 업그레이드)을 만들 때 참고하기 위함이다.

---

## 1. 핵심 명제

**컨텍스트는 상류에서 발원해 하류로 흐른다. 각 단계는 다음 단계가 읽을 *매체*에 그것을 남겨야 한다. 리뷰는 최하류라, 상류가 흘린 컨텍스트를 복원하려 애쓰는 자리다.**

```
도메인 설계 ──▶ 구현 ──▶ PR 작성 ──▶ 리뷰
   (왜 이 규칙)  (왜 이 코드)  (변경 의도)   (판단)
        │           │            │           │
   컨텍스트 발원 ───────────흐름───────────▶ 컨텍스트 소비
```

리뷰가 "코드 밖 Why"(제품·도메인 결정)를 코드만으로 복원할 수 없는 이유가 여기 있다. 그 Why는 **상류에서 생성되어 매체로 전달됐어야** 하는 것이지, 코드에서 유추할 수 있는 것이 아니다. 코드는 결정의 *결과*(필터 로직 등)만 담고, *왜 그 결정인지*는 담지 못한다.

## 2. 손실은 기본값, 보존이 능동 작업

각 전달 단계에서 컨텍스트는 **자연 증발한다** (엔트로피). 도메인 결정의 "왜"는 코드로 넘어오며 사라지고, 코드의 "왜"는 PR로 넘어오며 요약만 남고, PR의 뉘앙스는 리뷰어에게 닿을 때 또 깎인다. 적극적으로 잡지 않으면 무조건 샌다. **"컨텍스트 손실 최소화"의 핵심은 보존이 디폴트가 아니라 노력이라는 것.**

## 3. 캡처는 생성 시점이 가장 싸다 (shift-left)

컨텍스트를 잡는 비용은 **생성 시점에 가장 낮고**, 하류로 갈수록 폭증한다.

- 도메인 결정을 내리는 *그 순간*: "왜"가 작성자 머릿속에 생생함 → 한 줄 적으면 끝.
- 리뷰 시점에 복원: 추론·고고학 → 부정확하고 비쌈. 종종 불가능.

→ 파이프라인 상류일수록 캡처 ROI가 높다. 리뷰 스킬이 아무리 똑똑해도, 상류에서 안 잡은 컨텍스트를 싸게 복원할 방법은 없다.

## 4. 단계별 컨텍스트 종류와 매체

컨텍스트는 종류마다 적합한 매체가 다르다. 전부 한 곳(예: 주석)에 몰면 그 매체가 과포화된다.

| 단계 | 생성하는 컨텍스트 | 적합 매체 |
|---|---|---|
| 도메인 설계 | Why-도메인 (왜 이 규칙·이 모델) | 설계 문서 / ADR |
| 구현 | Why-구현 (왜 이 코드·이 구조) | 코드 주석 / 커밋 메시지 |
| PR 작성 | 변경 의도 (무엇을·왜 바꿨나, 도메인 Why 요약) | **PR 본문** |
| 리뷰 | (소비) + 판단 + 손실 진단 | 리뷰 코멘트 |

리뷰가 필요로 하는 "코드 밖 Why"의 1차 매체는 **PR 본문**이다. 그리고 PR 본문이 그것을 담으려면, 구현 단계에서 이미 그 컨텍스트를 (주석·커밋으로) 알고 있어야 하고, 그러려면 구현 시작 전에 도메인 의도가 명확해야 한다. 사슬이 위로 닿는다.

## 5. 질문 도구 = 백업 채널 + 환류 장치

파이프라인이 새는 지점에서, 인간(컨텍스트의 궁극 원천)에게 물어 공백을 채운다. 단:

- 질문은 **최후 수단**이다 — 비싸고(사용자 시간), 휘발성이다(기억이 흐려짐).
- 받은 답은 **쓰고 버리지 말고 매체로 되돌려 기록**해야 한다 (PR 본문·주석·문서). 환류하지 않으면 다음 사이클에서 같은 질문을 반복한다.

즉 질문 도구는 손실을 *그 자리에서* 메우는 동시에, *미래의 손실을 줄이는* 환류 루프의 일부여야 한다.

## 6. 리뷰는 손실을 진단하고 코멘트로 환류한다

리뷰의 산출물은 두 종류다:

1. **correctness 결함** — "이 코드가 틀렸다" (세부 구현, 에이전트가 잡음).
2. **컨텍스트 보존 결함** — "이 의도가 코드·PR 어디에도 없다" (예: "이 leftJoin의 의도가 명시 안 됨 → PR 본문에 추가 요망").

2번이 파이프라인 피드백의 구체적 실현이다. 리뷰가 상류의 손실 지점을 *진단*해 코멘트로 되돌리면, 다음 사이클(PR 템플릿·구현 관행)이 그 구멍을 막도록 개선된다. 리뷰는 파이프라인의 끝이자 *품질 게이트*다.

## 7. 리뷰 스킬(현재)의 위치와 책임

`yuumi:review`는 최하류 스킬이며, **상류 손실을 100% 복구할 수 없음을 스스로 안다.** 그래서:

- 코드로 복원 가능한 Why(🟢 기계적 연역 / 🟡 구조적 추론)는 복원하되 출처에 따라 단언/추정을 구분한다.
- 코드 밖 Why(🔴 의도)는 PR 본문·이슈·커밋에서 찾고, 없으면 confabulate하지 않고 사용자에게 묻는다.
- 어디에도 없으면 **"컨텍스트 공백"을 정직하게 표시**하고, 채운 답을 코멘트로 환류한다.

리뷰 스킬은 이 한계를 *결함*이 아니라 *설계*로 받아들인다 — 없는 걸 지어내는 게 아니라, 코드로 답할 수 있는 것과 없는 것의 경계를 정직하게 긋는 것이 핵심 기능이다.

## 8. 향후 스킬에의 함의 (PR, Implement)

이 파이프라인을 의식하고 만들면:

### Implement 스킬 (현재 것의 업그레이드)
- 구현 중 내린 결정의 **Why-구현을 주석/커밋에 능동 캡처** (생성 시점 = 최저 비용).
- 도메인 의도가 불명하면 **구현 전에 질문**으로 컨텍스트를 완성 (또는 사용자가 들고 옴).
- 캡처한 컨텍스트를 PR 스킬이 읽을 수 있는 형태(구조화된 커밋/주석)로 남김.

### PR 작성 스킬
- 구현 단계의 컨텍스트(주석·커밋·결정 로그)를 **PR 본문으로 승격** — 특히 코드에 안 남는 Why-도메인.
- 빠진 의도는 사용자 질문으로 채우고 본문에 기록 (환류).
- 리뷰 스킬이 찾을 위치(본문의 "왜" 섹션)에 명시적으로 배치.

### 세 스킬의 계약
컨텍스트는 **구현에서 시작해 리뷰까지 이어지는 단일 흐름**이다. 각 스킬은 (1) 자기 단계의 컨텍스트를 최대한 보존하고, (2) 다음 단계가 읽을 매체에 남기며, (3) 공백은 질문으로 채우되 그 답을 매체로 환류한다. 이 계약을 모든 스킬이 지키면, 사이클이 완성되지 않아도 *채워진 만큼* 리뷰가 매끄러워진다 (§9의 연속체).

## 9. 연속체 — 완성은 전제가 아니다

완벽한 사이클은 수많은 사람의 협업에서 "컨텍스트 손실 최소화" 시스템이 갖춰져야 *그나마* 가능하다. 따라서 목표는 "사이클 완성 후 리뷰"가 아니라, **파이프라인이 채워질수록 리뷰의 질문 횟수가 줄고 매끄러워지는 점진적 개선**이다.

- 리뷰 스킬 단독: 질문 도구로 공백을 메우며 동작 — 가치 있음.
- + Implement 스킬: 구현 컨텍스트가 보존됨 — 리뷰의 복원 부담 감소.
- + PR 스킬: 코드 밖 Why가 본문에 옴 — 리뷰가 "코드 밖"을 코드 밖에서 읽음.

각 스킬이 파이프라인 의식을 갖고 동작하면, all-or-nothing이 아니라 그라데이션으로 좋아진다.

## 부록: 이 문서가 `yuumi:review` SKILL.md에 주는 즉각 함의

1. Why 사슬의 **출처 라벨링**(🟢연역/🟡추론/🔴의도)과 **코드 경계 인식** — 경계를 넘으면 유추 중단, 출처 전환(PR/이슈→사용자).
2. **컨텍스트 공백 코멘트** — correctness 결함과 별개로, "이 의도가 어디에도 없음 → PR 본문에 명시 요망"을 리뷰 산출물의 한 종류로.
3. 질문으로 채운 답의 **환류** — 세션 내에서는 지도/코멘트에 기록 (영구화는 PR 스킬의 몫으로 이월).
